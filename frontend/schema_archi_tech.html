Schéma d'Architecture Technique

Client (Navigateur)

Point d'entrée de l'utilisateur. Il envoie des requêtes HTTP et affiche les pages HTML, CSS et JavaScript.

→

Serveur Nginx (Docker)

Reverse Proxy. Il reçoit toutes les requêtes. Il sert directement les fichiers statiques (front-end) et redirige les appels API vers le service back-end.

↔

Application Flask (Docker)

Le cœur de l'application. Reçoit les requêtes de Nginx, applique la logique métier et communique avec la base de données pour lire ou écrire des informations.

↔

Base de Données MongoDB (Docker)

Le système de stockage. Elle conserve toutes les données de l'application (chambres, utilisateurs, réservations) de manière persistante.

Explication de la "Machinerie"

Ce schéma illustre une architecture **multi-conteneurs découplée**, orchestrée par docker-compose.yml.

  • Découplage : Chaque service (Nginx, Flask, MongoDB) vit dans son propre conteneur isolé. On peut mettre à jour, redémarrer ou remplacer un service sans affecter les autres. C'est un gage de robustesse et de maintenabilité.
  • Sécurité : La base de données n'est pas accessible depuis l'extérieur. Seul le conteneur du back-end peut lui parler, via un réseau Docker privé. Nginx agit comme un bouclier, filtrant les requêtes.
  • Performance : Nginx, spécialisé dans le service de fichiers statiques, gère le trafic client efficacement, laissant l'application Flask se concentrer sur sa tâche principale : la logique métier.
